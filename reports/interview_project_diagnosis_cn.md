# Synerise 项目面试诊断与改造方案

## 一句话判断

这个项目已经达到“能讲、能跑、工程上不虚”的水平，但当前更像一个扎实的离线机器学习项目，还没有完全达到“大厂数据挖掘算法实习生项目”里那种任务定义锋利、实验设计完整、业务闭环明确的状态。

面试里它的主要加分项是：

- 数据真实且规模大，不是玩具数据
- 用了时间切分，不是随机切分
- 处理过大表和工程瓶颈，DuckDB on Parquet 这个点很好讲
- 不只是报 AUC，还做了错误分析和 Top-K 分析

面试里它最容易被质疑的地方是：

- 标签定义导致正例率过高
- 默认阈值下几乎把所有用户都判成 churn
- 模型和特征还偏 baseline 化
- 策略分析还没有走到 ROI 或 uplift 层面

## 面试里的项目定位

最稳的讲法不是“我做了一个推荐系统比赛项目”，而是：

“我基于 Synerise 官方电商行为日志做了一个用户级流失预测项目，重点不是刷榜，而是从大规模原始 parquet 日志出发，完成时间切分、用户特征构造、多模型对比和留存运营策略分析。”

这个定位的好处：

- 避开“你这个不是推荐系统排序”的追问
- 强调你做的是数据挖掘里非常典型的用户经营问题
- 自然引出工程、建模、策略三个层面的贡献

## 90 秒版本

我做的是一个用户级 churn prediction 项目，数据来自 Synerise RecSys 2025 官方原始电商日志。原始数据有 2 亿级 page visit，所以第一版用 pandas 聚合时遇到了明显内存瓶颈，后来我把特征工程改成了 DuckDB 直接扫 parquet 做 SQL 聚合。样本单位是 client_id，观察窗口 30 天，预测未来 14 天是否不再购买，并且严格按 snapshot 时间切分 train、valid、test，避免时间泄漏。

特征上我主要构造了多时间窗行为频次、recency、活跃天数、SKU 和类目多样性、价格统计、搜索长度和搜索后转化等特征。模型层面我对比了 Logistic Regression、Random Forest、LightGBM 和 XGBoost，最终 XGBoost 最好。除了离线指标，我还补了 active user 定义对比、错误分析和 Top-K 干预分析，让这个项目更接近真实留存运营场景。

## 3 分钟版本

### 1. 先讲业务问题

这个项目的目标不是做推荐排序，而是识别“当前用户未来 14 天会不会停止购买”，给运营提供优先干预名单。这个问题在互联网公司里更接近数据挖掘和用户经营场景，也更适合作为一个端到端项目来展示。

### 2. 再讲数据和工程难点

数据来自 Synerise 官方原始日志，包含 page visit、search、add to cart、remove from cart、product buy 和商品属性表。其中 page_visit 非常大，接近 2 亿行，所以我没有把它完整读进 pandas，而是改成 DuckDB on Parquet 直接在文件上做聚合，解决了大表处理的内存问题。

### 3. 再讲样本、标签和时间切分

我把样本单位定义成 client_id，以 snapshot 为锚点，取过去 30 天作为观察窗，未来 14 天是否没有 product_buy 作为 churn 标签。因为日志数据天然有时间顺序，所以我没有随机切分，而是按 snapshot 做时间切分，这样更接近真实上线场景，也能避免时间穿越。

### 4. 再讲特征和模型

特征主要是围绕用户行为强度、活跃度、转化意图和行为衰减速度展开，包括 1/3/7/14/30 天多窗口计数、最近一次行为距今天数、活跃天数、购买和加购的 SKU/类目多样性、价格统计，以及搜索长度、词数和搜索后 1 天内转化等特征。模型上我先用 LR 做线性 baseline，再用 RF、LightGBM、XGBoost 做对比。

### 5. 最后讲你不是只会跑模型

做完 baseline 后，我发现样本口径太宽会导致正例率很高，于是我补做了更严格的 active user 定义实验；另外我发现单纯调阈值收益不大，所以又补了 Top-K targeting 分析，把模型结果转成“如果运营只能触达前 5% 或 10% 高风险用户，该怎么圈人”的策略输出。这个过程说明我不是只停留在跑一个离线模型，而是在不断检查任务定义和业务落地方式是否合理。

## 面试官最可能追问的 8 个问题

### 1. 你为什么不做随机切分？

标准回答：

随机切分会把未来时段的行为模式泄漏给训练集，尤其是在日志场景里，同一个用户不同时间段的行为高度相关。这个项目是要用过去预测未来，所以时间切分比随机切分更合理，也更接近真实上线。

### 2. 你为什么说这个项目有工程难点？

标准回答：

因为 raw 数据里 page_visit 接近 2 亿行，直接 pandas 聚合在本机上不稳定，所以我把特征工程迁到了 DuckDB on Parquet，通过 SQL 直接做窗口聚合和特征物化。这个改动说明我有处理大规模行为日志的工程意识，而不是只会对小样本调模型。

### 3. 为什么最后选 XGBoost？

标准回答：

这个任务本质上是典型表格特征二分类问题，人工聚合特征比较丰富，XGBoost 对非线性和特征交互的建模能力更强，同时训练成本又可控，所以在当前设定下效果最好。

### 4. 你这个 PR-AUC 和 F1 很高，是不是任务太简单了？

标准回答：

是的，这也是我后来重点反思的地方。当前标签定义下正例率非常高，所以 PR-AUC 和默认阈值 F1 会偏乐观。也正因为这样，我后面才补了更严格的 active user 口径和 Top-K 分析，并且在面试里会更强调 AUC、任务定义和策略 lift，而不是单纯强调 0.97 的 PR-AUC。

### 5. 你为什么要补 active user 定义实验？

标准回答：

因为我发现如果只要求历史买过一次，样本会把很多已经长期沉默的历史用户也纳入进来，这会让 churn 标签过于容易。于是我增加了“观察窗内仍有关键行为”的更严格定义，用来验证样本口径变化对标签分布和模型效果的影响。

### 6. 这个项目最大的不足是什么？

标准回答：

最大的不足是任务定义还可以继续收紧，现在的标签让正例率太高，导致模型默认几乎会把所有用户判成 churn。第二个不足是特征和模型还偏 baseline，缺少更有数据挖掘味道的趋势特征、偏好漂移特征或更强的实验设计。

### 7. 如果继续做，你最想优化哪一块？

标准回答：

我会先优化任务定义和评估方案，再优化特征。因为如果标签和样本口径不够合理，后面再怎么调模型，提升都可能只是离线分数层面的。

### 8. 这个项目怎么跟业务落地接起来？

标准回答：

我会把它作为一个 retention targeting 模型，而不是简单的二分类器。业务上通常不是对所有用户干预，而是基于预算约束触达前 1%、5%、10% 的高风险用户，所以我会重点输出 Top-K 名单、分层特征和分群策略，而不是只给一个 0.5 阈值下的标签。

## 当前不足清单

### P0. 标签定义太宽，导致正例率过高

问题：

- 现在的 churn 定义是未来 14 天没有任何购买就算流失
- 对很多低频购买用户来说，这个定义会把“暂时没买”也当成 churn
- 结果是正例率非常高，指标解释空间变差

面试影响：

- 很容易被面试官追问“这是不是个几乎全是正样本的问题”
- 容易让高 PR-AUC、高 F1 的说服力下降

### P0. 阈值下几乎把所有用户都判成 churn

问题：

- 默认阈值 0.5 下，目标用户覆盖接近全量
- 说明模型的二分类输出不够适合直接当运营规则使用

面试影响：

- 如果你只讲 Precision / Recall / F1，面试官会觉得这个结果不够实用
- 必须主动把叙事切到 ranking / Top-K / targeting

### P1. 算法亮点不足

问题：

- 现在的特征大多是标准人工聚合
- 模型也主要是四个传统 baseline
- 这更像一个完整的数据挖掘 baseline，而不是一个带明显算法创新点的项目

面试影响：

- 能证明你会做项目
- 但还不够证明你有更强的算法设计能力

### P1. 实验设计还不够强

问题：

- 当前主要是一次性时间切分
- 缺少 rolling backtest
- 缺少按 snapshot、按用户分层、按价值层的稳定性比较

面试影响：

- 面试官会担心你结果依赖某一段时间窗口
- 也会追问线上泛化能力

### P1. 业务闭环不完整

问题：

- 现在有 Top-K 分析，但没有成本约束、ROI、uplift 或 A/B 设计
- 所以项目更像“风险识别”，还不是“经营策略优化”

面试影响：

- 对算法面试已经够用
- 但对更偏业务分析或策略挖掘的面试，会显得收尾不够完整

### P2. 工程复现性还有瑕疵

问题：

- 运行时依赖需要补全
- 还缺少最小测试集和自动化校验

面试影响：

- 不是核心致命点
- 但会影响项目的专业感

## 最值得补的 3 个升级项

### 升级 1. 重新定义 churn 任务

目标：

把“未来 14 天没买”改成更贴近业务和购买周期的流失定义。

建议做法：

- 只在最近 60 天有购买的用户中定义 churn
- 或者要求观察窗内有购买/高意图行为，再看未来是否完全沉默
- 或者按历史购买周期分层定义流失

预期收益：

- 降低正例率
- 提高任务难度和指标含金量
- 让 Top-K lift 更有解释力

面试价值：

这会明显增强你对标签定义和业务问题建模的理解力。

### 升级 2. 补趋势和偏好漂移特征

目标：

让特征不只是静态聚合，而是体现用户行为变化趋势。

建议做法：

- 7 天行为 / 30 天行为的比值
- 最近 7 天与前 7 天的行为差分
- 用户最近购买类目与历史偏好的偏移程度
- 搜索到加购、加购到购买的阶段式衰减特征

预期收益：

- 提高对“将要流失”的早期识别能力
- 比单纯频次统计更像数据挖掘项目

面试价值：

这个升级最能体现“不是堆特征，而是在建模行为变化”。

### 升级 3. 把评估做成 rolling backtest

目标：

验证模型在多个时间段上的稳定性，而不是只看一次切分结果。

建议做法：

- 用更早的 snapshot 训练
- 在后续多个 snapshot 上滚动验证
- 输出每个 snapshot 的 AUC、PR-AUC、Top-K lift

预期收益：

- 更真实评估模型时序泛化能力
- 能发现某段时间结果是否异常好或异常差

面试价值：

这个点会明显提升项目的“严谨度”，很适合回答“你怎么确保模型不是偶然有效”的问题。

## 一周改造优先级

如果时间只有一周，建议按下面顺序补：

1. 先补新的 active user / churn 定义实验
2. 再补趋势特征
3. 最后补 rolling backtest 和分 snapshot 指标

原因：

- 第一步决定任务有没有意义
- 第二步决定模型有没有更强表达力
- 第三步决定结果有没有说服力

## 面试时该主动承认什么

建议你主动承认两件事：

- 这个项目的最大问题是当前 churn 定义偏宽，导致正例率过高
- 我后续的优化方向不是盲目刷模型，而是先收紧任务定义，再增强时序特征和评估设计

这种讲法不会减分，反而会加分，因为它说明你不是沉迷离线分数，而是在反思问题设定本身。

## 面试时不要这样讲

不要讲：

- “我这个项目 F1 很高，所以模型很好”
- “我把四个模型都跑了，所以很完整”
- “PR-AUC 0.97，效果非常强”

更好的讲法：

- “当前设定下默认分类指标偏乐观，所以我更重视 AUC、Top-K lift 和任务定义合理性”
- “我先用标准 baseline 建了一个稳定闭环，再通过 active user 定义和策略分析检查这个问题是否真的成立”
- “这个项目最大的价值不只是分数，而是把大规模日志建模和业务解释串起来”

建议写法一：

基于 Synerise 官方电商行为日志构建用户级 14 天流失预测模型，处理 2 亿级行为事件，完成从原始 parquet 到特征聚合、时间切分、多模型训练与策略分析的全流程。

建议写法二：

使用 DuckDB on Parquet 重构大规模日志特征工程，构建多时间窗行为、活跃度、转化与搜索特征，并基于时间切分对比 LR、RF、LightGBM、XGBoost，进一步补充 active user 定义对比与 Top-K 留存干预分析。

## 最后结论

这个项目可以拿去面试，而且中上水平。

如果按 10 分制粗略打分：

- 工程完整度：8
- 业务理解：7.5
- 建模基础：7.5
- 算法亮点：6
- 实验严谨性：6.5
- 面试可讲性：8

如果你把“任务定义重做 + 趋势特征 + rolling backtest”这三件事补上，它会从“不错的实习项目”明显提升到“很像大厂算法实习生认真做过的项目”。
