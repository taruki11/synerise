# Synerise 项目面试讲述稿

## 30 秒版本

我做的是一个用户级 14 天流失预测项目，数据来自官方 Synerise RecSys 2025 raw dataset。我把 6 个月、2.25 亿级电商行为日志按 `client_id` 聚合成 67 个用户特征，严格用时间切分训练了 Logistic Regression、Random Forest、XGBoost 和 LightGBM，最终最好的是 XGBoost，在更严格 active user 口径下测试集 AUC 是 0.7066。项目里我还专门做了 active user 定义对比和 Top-K 运营策略分析，让结果更像真实留存建模场景。

## 1 分钟版本

这个项目的目标不是做推荐，而是预测当前活跃用户未来 14 天会不会停止购买。样本单位是 `client_id`，观察窗 30 天，预测窗 14 天。原始数据里有 `page_visit`、`search_query`、`add_to_cart`、`remove_from_cart`、`product_buy`，再加 `product_properties` 商品属性表。

项目的核心工作主要有三块。第一块是大数据特征工程，我做了 1/3/7/14/30 天多时间窗行为频次、最近一次行为距今天数、漏斗转化率、活跃天数、去重 SKU 数、品类数、价格聚合、搜索后转化等特征。第二块是工程落地，因为 `page_visit` 有 1.99 亿行，pandas 直接读会爆内存，所以我把全流程改成 DuckDB on Parquet。第三块是建模和评估，我严格按时间切分 train/valid/test，最终 XGBoost 效果最好。

后来我又补了两个更像业务项目的实验。第一个是把 active user 定义从“历史买过”收紧成“历史买过且观察窗内还有行为”，这样样本从 4,281,892 降到 2,423,390，正例率从 94.61% 降到 91.10%，更容易解释也更贴近运营场景。第二个是做阈值和 Top-K 分析，结果发现最优阈值只是从 0.50 到 0.45，提升很小，但 Top 10% 高风险用户能覆盖 10.57% 的真实流失用户，所以我会更强调 Top-K 策略。

## 3 分钟版本

### 1. 业务问题怎么定义

- 电商场景里，比起推荐排序，这个项目更偏增长和经营分析。
- 我预测的是“当前活跃用户未来 14 天是否停止购买”，方便运营提前做召回。
- 样本单位是 `client_id`，不是 session，也不是商品。

### 2. 数据和工程难点怎么讲

- 这套数据不是教学数据，行为事件总量大约 225,224,262 条。
- 最大的一张 `page_visit.parquet` 有 199,451,980 行，直接 pandas 会有明显内存压力。
- 所以我把特征聚合改成 DuckDB SQL 直接扫 parquet，在本机上也能稳定跑完整项目。

### 3. 特征工程怎么讲

- 我把用户最近 1/3/7/14/30 天五类行为都做了频次聚合。
- 再做 recency、活跃天数、购买天数、去重 SKU 数、去重类目数、价格统计和漏斗比率。
- 由于 raw 数据里的搜索字段是 query string，不是 embedding，所以搜索特征改成了查询次数、查询长度、词数，以及搜索后 1 天内加购/购买次数。
### 4. 模型和结果怎么讲

- 我先用 LR 做线性 baseline，因为它可解释、稳定，适合做对照。
- RF 用来验证树模型是否比线性模型更能吃到非线性特征。
- XGBoost 和 LightGBM 是主力表格模型，最终 XGBoost 效果最好。
- 在 baseline 口径下，XGBoost test_refit AUC 是 0.7330；在更严格口径下 AUC 是 0.7066。

### 5. 两个新增实验怎么讲

- Active user 定义对比：我不是默认接受最初样本定义，而是主动验证“历史买过”和“最近真的活跃”两种口径对标签分布和模型结果的影响。
- 阈值和 Top-K：我发现这个任务下阈值微调收益不大，所以进一步把模型结果转成运营可执行的 Top-K 圈选策略。
- 这两步会让项目从“会跑模型”变成“会把模型接到业务动作上”。

## 新增实验的推荐答法

### 实验一：更严格 active user 定义

我后来补了一个 active user 定义对比实验。最开始的口径是用户在 snapshot 之前历史上买过一次就纳入样本，这个定义比较接近官方 churn framing，但我担心它会把一些已经很久没动静的用户也放进来，导致流失正例率过高。所以我又加了更严格的版本，要求用户不仅历史上买过，而且观察窗 30 天内至少有一次 `page_visit`、`add_to_cart` 或 `product_buy`。结果样本量从 4,281,892 下降到 2,423,390，正例率从 94.61% 降到 91.10%。虽然 AUC 从 0.7330 降到 0.7066，但是这个口径更符合“当前活跃用户”的业务定义，也更容易向面试官解释标签分布。

### 实验二：阈值优化和 Top-K 策略

我还做了阈值优化和 Top-K 分析。验证集按 F1 搜索后，最优阈值从 0.50 只移动到 0.45，提升很小，说明这个任务下单纯改阈值收益有限。所以我更重视 Top-K 圈选，因为现实里运营资源通常有限，不会对所有用户都触达。在更严格口径下，XGBoost 测试集 Top 10% 高风险用户的 churn precision 是 0.9630，可以覆盖 10.57% 的真实流失用户，lift 是 1.0570。这就能把模型结果翻译成一句很业务的话：如果只触达分数最高的 10% 用户，能优先覆盖一部分真实流失人群。

## 面试里最值得主动强调的 6 个点

- 这是 2 亿级行为日志，不是普通表格竞赛。
- 我用了时间切分，没有随机切分。
- 我处理了真实 raw data 字段差异，而不是照着预想 schema 写代码。
- 我把 pandas 管道改成 DuckDB on Parquet，解决了大表内存问题。
- 我不只报 AUC，还补了 active user 口径对比、阈值优化和 Top-K 运营策略。
- 我能解释为什么更严格定义下 AUC 略降，但业务解释性更强。

## 结束句模板

如果让我总结这个项目，我会说：我不仅把官方原始行为日志跑通了，还把 churn 建模从“技术上可训练”推进到了“业务上可解释、可落地圈选”的版本。
